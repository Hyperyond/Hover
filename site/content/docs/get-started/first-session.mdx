# Your first session

A guided walkthrough of the Hover chat and views. Estimated time: 5 minutes.

## Prerequisites

- Followed [Quick start](/docs/get-started/quick-start/) — the Hover extension is installed and your dev server is running.
- The **Hover** icon is visible in the VS Code Activity Bar.
- A debug Chrome on port 9222 (Hover prompts you to launch it on the first run).

## Step 1 — Open the chat

Click the **Hover** icon in the Activity Bar to open the chat webview.

You'll see:
- A header pill `claude ▾` showing the active coding agent
- A mode switch — Flow / 🟢 QA Testing (QA carries API + penetration toggles)
- An empty conversation body
- A composer with a textarea, a model pill, a voice (speech) toggle, and **Send**

Alongside the chat, the `hover` container has these views: **Dashboard** (your saved `__vibe_tests__/` specs, folder-grouped, with pass/fail and ✨ Optimize), **Conversations** (one record per agent run — agent, model, cost, outcome), **Network** (captured API calls; shown when QA's API / penetration toggles are on), and **Environments** (Local + remote targets and their accounts). A **Settings** view covers agent / model / browser visibility / speech / model API key.

## Step 2 — Set up an environment (optional)

Open the **Environments** view. The active environment drives the run target URL; remote targets skip the dev-server spawn entirely.

Define a test account on an environment, then in chat just mention `@account` — the agent logs in for you. Credentials are parameterized into `process.env` references: never written into the spec, the JSDoc, or the sidecar. The same names export to your CI secrets in one click. (Account passwords are kept in VS Code SecretStorage, never in the committed roster.)

## Step 3 — Send your first prompt

Type a natural-language instruction describing a flow you want to verify on your app. For example:

```
log in with @account, navigate to the dashboard, and verify
the user's name appears in the header
```

```
add a product to cart, go to checkout, verify the total is correct
```

```
submit the contact form with empty fields, verify each required-field
error appears
```

Press `↵` or click **Send**. The button switches to **Stop** while the agent runs.



Press `↵` or click **Send** to submit.



## Step 4 — Watch the agent drive

Events stream into the chat as the agent works. Tool steps fold under AI-narration titles:
- `Opening page` / `Clicking …` / `Filling form` — tool calls, grouped under each natural-language intent
- A spinner on the currently-running step
- Running cost ($) ticks in the header

The agent's verification report renders as a Result card at the end. If it found bugs, a Findings card appears alongside, severity-coloured.

## Step 5 — Save the verified session

When the run finishes, Hover asks **Save this run as a spec?** in the composer. Confirm and it writes `__vibe_tests__/<slug>.spec.ts` using `getByRole` / `getByLabel` / `getByTestId`.

The extension's Save emits **only** the Playwright spec. It appears in the **Dashboard**, where you can also ✨ **Optimize** it (an optional AI pass you accept via diff — the original is always kept).

## Step 6 — Run the spec without Hover

The saved file is a plain `@playwright/test` spec. Run it like any other Playwright test in your project:

```bash
npx playwright test __vibe_tests__/<slug>.spec.ts
```

This is the point of crystallization: the AI authored the test once, but the saved spec runs forever in CI with zero agent involvement.
