# Install

Hover is a **VS Code extension**. There's no bundler plugin to install and nothing to add to your project — the extension drives your existing dev server over CDP regardless of framework.

## Install the VS Code extension

Install it from the **[VS Code Marketplace](https://marketplace.visualstudio.com/items?itemName=hyperyond.hover-dev)**:

1. Open the Extensions view (`⇧⌘X` / `Ctrl+Shift+X`), search **`hover-dev`**, and click **Install** — or run `code --install-extension hyperyond.hover-dev`.
2. That's it. (Building from source? `pnpm --filter hover-dev package` produces a `hover-dev-<version>.vsix` you can sideload via the `⋯` menu → **Install from VSIX…**.)

That's the whole install for Hover itself. The extension ships the engine (`@hover-dev/core`) inside it and runs it under VS Code's own Node — no global service, no per-project plugin, no bundler config to edit.

## Before you start

You need two things:

1. **VS Code** (or a VS Code-compatible editor) with the Hover extension installed.
2. **One coding-agent CLI installed and logged in.** Hover spawns whichever you have on `PATH`; it doesn't bundle an LLM and doesn't take API keys.

<Callout type="tip" title="Don't have an agent CLI yet?">

- **Claude Code** — `npm install -g @anthropic-ai/claude-code`, then `claude login`. Uses your Claude Pro / Max subscription.
- **OpenAI Codex** — `npm install -g @openai/codex`, then `codex login`. Uses your ChatGPT plan.

Either one works. You can switch between agents from the chat header or Settings any time.

</Callout>

That's the only thing to configure — Hover ships no model SDK and no keys of its own. (Prefer a model API key over your CLI subscription? See [Use an API key](/docs/get-started/api-key/).)

## Use it

Open your project in VS Code, start your dev server the way you already do (`pnpm dev` / `npm run dev` / …), then open the **Hover** chat from the Activity Bar and send a prompt. On the first run, Hover prompts you to launch an isolated debug Chrome on port 9222 (a clean temp profile — never your everyday browser). See [Your first session](/docs/get-started/first-session/) for the full walkthrough.

Remote / staging targets, multi-environment accounts, and per-environment credentials are managed in the **Environments** view — see [Your first session](/docs/get-started/first-session/).

## Troubleshooting

### `EADDRINUSE: address already in use :::5173` (or 5183, etc.)

Something else is on the port. Most often: a prior `next dev` / `vite` that didn't shut down cleanly. Kill it:

```bash
lsof -i :5173 -t | xargs kill         # replace 5173 with your dev port
```

If `lsof` shows a stale Hover engine on 51789–51798 too, kill those: `lsof -i :51789 -t | xargs kill`.

### The chat can't reach a debug Chrome

On first use Hover offers to launch an isolated debug Chrome on port 9222 (a clean temp profile under `<tmpdir>/hover-chrome` — not your everyday browser). If a debug Chrome is already running but you're not in it, Hover brings the right window to the front. If 9222 is occupied by something else, free it: `lsof -i :9222 -t | xargs kill`.
